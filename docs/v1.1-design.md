# v1.1 Design Doc: Audit log, Onboarding, Public Ingress

Status: **DRAFT — for design review**
Author: Nexus team
Last updated: 2026-06-22

## 1. Background

v1 (shipped) already has:
- Console (admin + self-service) at `console.tail7d361a.ts.net` (Tailscale only)
- Marketing site at `nexus.ffx.ai` (Cloudflare Pages, just launched)
- SSO via OIDC (Keycloak in prod)
- Strict BYOK, virtual keys, eval/spend tracking

v1.1 fills three gaps that block a public-ish launch (where users outside the
Tailscale network can sign up via the marketing site and reach the console):

| # | Gap | Why it matters |
|---|---|---|
| 1 | **Audit log** | Today only `sso.login` and `credential.delete` are audited. Admin actions (user create/delete, key create/revoke, credential create/rotate) are not recorded. Compliance / incident response gap. |
| 2 | **Onboarding flow** | Self-service signup exists, but post-signup the new user lands on an empty Account page with no guidance. Conversion / activation risk. |
| 3 | **`app.nexus.ffx.ai` public ingress** | Console today is only reachable via Tailscale. The marketing → "Login" flow currently has no public destination. Blocks the natural "I clicked Sign up on the marketing site" path. |

This document scopes v1.1 to those three workstreams. The full roadmap lives
in `docs/enterprise-model.md`; v2 work (platform credentials, customer IdP,
multi-tenant data planes) is explicitly out of scope.

## 2. Workstream A — Audit log (depth: complete what's started)

### 2.1 Current state

- **Table**: `audit_log(org_id, action, target_id, detail, created_at)` exists in Postgres
  (see `internal/core/store.go:423`).
- **API**: `Store.Audit(ctx, orgID, action, targetID, detail)` is best-effort
  (errors swallowed). No reader, no console endpoint.
- **Coverage**: Only two call sites —
  - `internal/console/sso.go:221` — `sso.login`
  - `internal/core/store.go:418` — `credential.delete`

  **Every other admin/member action is silent.** That is the gap.

### 2.2 v1.1 goals

1. **Coverage** — every state-changing action writes an audit entry. List:
   - `auth.login` (email + password)
   - `auth.logout`
   - `user.create` (admin)
   - `user.delete` (admin)
   - `key.create` (member / org)
   - `key.revoke`
   - `credential.create`
   - `credential.rotate`
   - `credential.delete` (already)
   - `sso.login` (already)
   - `me.update` (member — e.g. enforce_limits toggle)
   - `enforce_limits.toggle` if we want it granular

2. **API** — `GET /api/audit` (admin only) with:
   - `?limit=` (default 50, max 500)
   - `?action=` (filter)
   - `?user_id=` (filter; resolved to email)
   - `?since=` (RFC3339 or duration, e.g. `24h`)
   - Returns rows ordered by `created_at DESC`.

3. **UI** — Admin-only "Audit" tab in the console showing a table of recent
   entries (action, actor email, target, time, detail). New React component
   in `web/src/Account.tsx` (or a new `Audit.tsx`).

4. **Retention** — Documented, not enforced. v1.1 ships with "keep forever
   in Postgres"; a future migration will add a TTL job. Note this as a
   follow-up, not v1.1 scope.

### 2.3 Non-goals (v1.1)

- Exporting audit log (CSV / JSON). v2.
- Streaming audit events to an external SIEM. v2.
- Per-actor ip_address / user_agent capture. Schema would need a migration;
  out of scope.
- Immutable / append-only guarantees. Postgres trust model is enough for v1.1.
- Audit UI for non-admin users (every audit entry is org-scoped already; only
  admins need to *browse* it).

### 2.4 Implementation outline

**`internal/core/store.go`** — add:

```go
type AuditEntry struct {
    ID         string
    OrgID      string
    ActorID    string  // user_id of caller, empty if system
    Action     string
    TargetID   string
    Detail     string
    CreatedAt  time.Time
}

// ListAudit reads the most recent entries for an org.
func (s *Store) ListAudit(ctx context.Context, orgID string, limit int) ([]AuditEntry, error)
```

Schema migration: add `actor_id` column to `audit_log` (nullable; old rows
have NULL).

**`internal/console/admin.go`** — new handler:

```go
r.Get("/audit", s.requireAdmin(s.listAudit))
```

Calls `s.store.ListAudit(...)` and serializes.

**`internal/console/{auth,server}.go`** — add `s.audit(...)` calls to the
handlers listed in §2.2. The audit wrapper already exists in `server.go:297`
but does not pass `actor_id`. Extend it:

```go
func (s *Server) audit(ctx context.Context, orgID, actorID, action, targetID, detail string) { ... }
```

Use the caller's `u.ID` as `actorID`. The wrapper lives next to the existing
one; update all call sites.

**`web/src/Audit.tsx`** — new component. Table with columns:
Time | Actor | Action | Target | Detail. Sort by time desc. Empty state:
"No audit events recorded yet." Admin role only — guarded client-side by
checking `/api/me` role and redirecting to `/account` if `role !== 'admin'`.

**Tests**:
- `internal/console/admin_test.go` — add `TestListAudit` happy path + 403
  for non-admin.
- `internal/core/store_test.go` — add `TestAuditList` round-trip.

### 2.5 Rollout

- One PR per layer is fine, but a single PR for "audit coverage + API + UI"
  is also OK. Suggested: one PR with backend (migration, store, admin.go)
  + one PR with UI.
- Migration is forward-compatible: existing rows have `actor_id = NULL`.
- Deploy order: backend PR → smoke `GET /api/audit` returns rows → UI PR.

## 3. Workstream B — Onboarding flow (depth: lightweight)

### 3.1 Current state

`/api/auth/register` accepts email, password, **optional** `provider`,
`provider_secret`, `key_name`. If both provider and secret are present, the
backend also creates the credential and a default virtual key in one call.
The console (`web/src/Account.tsx`) shows the resulting state, but there is
**no step-by-step guidance** for a user who signs up with just email + password.

The new user's first 30 seconds:
1. Lands on `/account`
2. Sees "0 virtual keys" and "0 credentials"
3. Reads Account page help text
4. Has to figure out they need to (a) register a provider credential, (b) create a virtual key, (c) point their SDK at the gateway URL

That is fine for engineers. It is **not** fine for product managers, designers,
or anyone evaluating Nexus for the first time.

### 3.2 v1.1 goals

1. **First-run banner** on `/account` when the user has 0 credentials and
   0 keys. CTA: "Set up your first provider key" → opens a modal or routes
   to `/account?action=add-credential`. Persist a `onboarded_at` timestamp
   on first successful credential create to hide the banner.

2. **Step indicator** in the Account page sidebar for users with
   `onboarded_at IS NULL`:
   - [✓] Create account
   - [ ] Add a provider credential
   - [ ] Create a virtual key
   - [ ] Make your first chat completion

3. **Inline help** on each section ("What is a provider credential?",
   "What is a virtual key?") — one-paragraph tooltips. Already partially
   present in `Account.tsx`; expand.

4. **Sample curl / OpenAI SDK snippet** in the "Virtual key" section that
   copies with a click. Pre-filled with the user's actual gateway URL
   (`window.location.origin` for prod; Tailscale URL fallback in dev).

5. **Welcome email** — **out of scope for v1.1.** We don't have an email
   service wired up; that lands in v2 (see enterprise-model §4).

### 3.3 Non-goals (v1.1)

- Per-step telemetry (which step did users drop off at). Would need a
  product-analytics service. v2 or later.
- Interactive product tour (e.g. Intro.js / Shepherd). Heavier, can wait.
- "Invite teammates" flow. v2 (multi-tenant).
- Onboarding for SSO users (post-OIDC, same flow applies but no password
  step). Out of scope — they land in the same banner experience.

### 3.4 Implementation outline

**Backend**: trivial.

- Migration: add `onboarded_at TIMESTAMPTZ` column to `users` table.
- `Store.MarkOnboarded(ctx, userID) error` — set `onboarded_at = NOW()`.
- Call it from `createMyCredential` (success path) — first credential = done.
- Add `OnboardedAt *time.Time` to `core.User` struct, expose in `/api/me`.

**Frontend**: `web/src/Account.tsx` changes only.

- New sub-component `<OnboardingChecklist />` that:
  - Calls `GET /api/me`, checks `onboarded_at`
  - Renders banner + 3-step list if not onboarded
  - Hidden for admins (or admin gets a different view — TBD)
- New sub-component `<ProviderHelp />` and `<VirtualKeyHelp />` — collapsible
  help text with `?` icon.
- New sub-component `<CodeSnippet gatewayUrl={...} virtualKey={...}>` —
  renders a `<pre>` with a "Copy" button.

**Tests**:
- `internal/console/auth_test.go` — extend register test to assert
  `onboarded_at` is set after credential create.
- Frontend: existing Account tests cover the new sub-components; no
  dedicated test file required if Account.tsx is the only changed file.

### 3.5 Rollout

- One PR.
- Roll out behind a feature flag `NEXUS_ONBOARDING_V11` (env var) so it can
  be disabled if the banner is annoying. Default ON.
- After 1 week of prod, review user feedback. If banner click-through > 50%,
  keep. If <10%, iterate copy or remove.

## 4. Workstream C — `app.nexus.ffx.ai` public ingress

### 4.1 Current state

- Gateway: `https://nexus.tail7d361a.ts.net` (Tailscale)
- Console: `https://console.tail7d361a.ts.net` (Tailscale)
- Both are protected by Tailscale auth (only Tailscale members reach them).
- Marketing: `https://nexus.ffx.ai` (public, Cloudflare Pages).

The marketing site currently has no "Log in" link that resolves to the
console — because the console isn't reachable from a public browser.

### 4.2 v1.1 goals

1. **Expose the console publicly** at `https://app.nexus.ffx.ai`. Two options:

   | Option | Pro | Con |
   |---|---|---|
   | A. Public ingress via Cloudflare + Tailscale funnel | Single domain for users. No new infra. | Funnel is a Tailscale feature; depends on Tailscale staying in the stack. |
   | B. Public ingress via Cloudflare Tunnel (`cloudflared`) on the cluster | Decouples from Tailscale. Production-grade. | More moving parts. Requires running `cloudflared` in `tenant-nexus`. |

   **Recommend B.** `cloudflared` is already a dependency-free binary, has
   mature access controls, and removes the Tailscale dependency from the
   user-facing path.

2. **DNS**: `app.ffx.ai` CNAME → `<random>.cfargotunnel.com` (Cloudflare
   Tunnel endpoint). Add a CNAME proxy in Cloudflare Dashboard.

3. **Auth on the new ingress**:
   - Console requires login (session cookie). All `/api/*` routes are
     guarded by `requireUser` / `requireAdmin`. The SPA itself returns
     `index.html` for all paths (existing `spaHandler`).
   - No IP allowlist — anyone can hit `app.nexus.ffx.ai` but they see
     the login form until they authenticate.
   - SSO users (Keycloak) will be the primary login path going forward.
   - Self-signup is gated by `allowSignup` (config flag, default ON for
     OSS; admins can disable per-deployment).

4. **Headers**:
   - `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`
   - `X-Frame-Options: DENY`
   - `X-Content-Type-Options: nosniff`
   - `Referrer-Policy: strict-origin-when-cross-origin`
   - `Content-Security-Policy: default-src 'self'; connect-src 'self' https://nexus.ffx.ai https://app.nexus.ffx.ai wss://app.nexus.ffx.ai; style-src 'self' 'unsafe-inline'; script-src 'self'; img-src 'self' data:; font-src 'self' data:`

5. **Rate limits on the new ingress**:
   - Anonymous (login page, /api/auth/login, /api/auth/register): 30 req/min/IP
   - Authenticated: existing per-user RPM limits

6. **Logout / revoke**:
   - Local logout (`DELETE /api/auth/session`) clears the session cookie.
   - For SSO: `end_session_endpoint` from the OIDC provider (Keycloak) is
     called on logout so the user is logged out of the IdP too.

### 4.3 Non-goals (v1.1)

- Magic-link / passwordless login. v2.
- CAPTCHA on the signup form. Low priority; revisit if abuse appears.
- 2FA / WebAuthn. v2.
- Per-IP abuse detection. v2.
- The gateway (`v1/chat/completions`) is **not** publicly exposed by this
  workstream. That happens in a later v1.1.x if at all; the gateway is
  typically used by trusted services with virtual keys, not by end users.

### 4.4 Implementation outline

**`deploy/cozystack/`** — add `07-cloudflared.yaml` (Deployment + Service +
Secret + Ingress). It:
- Mounts a Tunnel token (K8s secret) created via `cloudflared tunnel create`.
- Configures the tunnel to forward `app.nexus.ffx.ai` → `nexus:8081` (in-cluster
  console service).
- Runs as a non-root user (securityContext `runAsNonRoot: true`,
  `allowPrivilegeEscalation: false`, `capabilities.drop: [ALL]`).

**Cloudflare side**:
- Create tunnel: `cloudflared tunnel create nexus-app`
- Add CNAME: `app.ffx.ai` → `<tunnel-id>.cfargotunnel.com` (DNS-only at first
  for verification, then proxied).
- Add ingress rule: `app.nexus.ffx.ai` → `http://nexus.tenant-nexus.svc.cluster.local:8081`

**`internal/console/server.go`** — add a middleware that sets the security
headers (above). Apply to the chi router globally, **before** `withUser`
so even unauthenticated requests get them.

**`internal/limiter/`** — add per-IP rate limit for the auth routes. The
existing limiter is per-virtual-key / per-user; we need a separate, smaller
limiter keyed on `(route, ip)`. Implementation:
- New `Limiter` instance with `Capacity: 30, Refill: 30/min`.
- Wrap `/api/auth/login`, `/api/auth/register`, `/api/auth/sso/*` in the
  new middleware.
- Source IP: `r.Header.Get("CF-Connecting-IP")` if behind Cloudflare,
  else `r.RemoteAddr` (split on `:`).

**`scripts/test_prod_smoke.sh`** — extend with `app.nexus.ffx.ai` checks
once the DNS resolves. Manual for the first 24h while certs propagate.

**`docs/cloudflare-tunnel.md`** — new doc. Setup steps for `cloudflared`,
Tunnel creation, DNS, ingress rules, token rotation.

### 4.5 Rollout

- **Phase 1 (no DNS yet)**: deploy `cloudflared` to the cluster, run
  tunnel, verify it forwards locally (`kubectl port-forward` + tunnel).
- **Phase 2 (DNS, internal)**: add CNAME in Cloudflare with proxy OFF.
  Verify from prod network. Inspect headers, rate limits, auth flow.
- **Phase 3 (public)**: turn proxy ON. Announce.
- Phase 1 + 2 are pre-launch; Phase 3 is the public launch.

This is a **3-PR sequence**:
1. PR-1: `cloudflared` Deployment + ClusterIP service + secret. (Internal only.)
2. PR-2: Security headers + per-IP rate limit + tests.
3. PR-3: Public DNS (Cloudflare CNAME proxy ON) + `app.nexus.ffx.ai` verification.

## 5. Cross-cutting

### 5.1 Dependencies

- **Audit log**: none (uses existing Postgres). Shipped first.
- **Onboarding**: depends on audit log migration for `actor_id` if we want
  to audit `me.update` from onboarding flow. Otherwise independent.
- **Public ingress**: depends on `cloudflared` getting deployed; the
  `CloudflareAccount` secret. No Nexus code dependency.

### 5.2 Order of operations

```
[1] Audit log backend (PR 1)        ──┐
[2] Audit log UI (PR 2)               ├──> shipped together as v1.1-alpha
                                     │
[3] Onboarding (PR 3)             ──┘
                                     │
[4] cloudflared deploy (PR 4)        ──┐
[5] headers + rate limit (PR 5)        ├──> shipped as v1.1-beta
[6] public DNS (PR 6)                ──┘
```

### 5.3 v1.1 acceptance criteria

- [ ] `GET /api/audit` returns the last 50 audit events, admin-only, 403 for
      non-admins. UI renders the table.
- [ ] `Store.Audit` is called from every state-changing handler (auth,
      user, key, credential, me). Verified by code review checklist.
- [ ] First-time user sees an onboarding banner; banner disappears after
      the first credential is created.
- [ ] `app.nexus.ffx.ai` returns 200 over HTTPS with the security headers
      from §4.2 set.
- [ ] Anonymous login attempts are rate-limited (30/min/IP). Test
      script verifies 31st request returns 429.
- [ ] Existing prod smoke (`scripts/test_prod_smoke.sh`) still passes.

### 5.4 Out of scope for v1.1 (deferred to v2+)

- Per-org rate limit (currently global per-user).
- Audit log UI filters (date range picker, multi-select action filter).
- Webhook for audit events (Slack, SIEM).
- Per-actor IP capture.
- Magic link signup, 2FA.
- Multi-org / multi-tenant data plane (see enterprise-model §2).

## 6. Open questions

1. **Audit log UI placement**: separate `/audit` route, or fold into
   the existing `/account` admin tab? (Lean: separate route, with a tab
   in Account for non-admins that says "Admins can view audit logs at /audit".)
2. **Onboarding banner copy**: English only for v1.1, or also Korean? (Lean:
   English only; i18n is a separate workstream.)
3. **Public ingress rollout window**: low-traffic weekend vs. weekday morning?
   (Lean: weekday morning so we can monitor during business hours.)
4. **Per-IP rate limit scope**: only auth routes, or all unauthenticated traffic?
   (Lean: only auth routes. `/healthz` etc. shouldn't be rate-limited.)
