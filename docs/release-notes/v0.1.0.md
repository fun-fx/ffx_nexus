# v0.1.0 pilot handoff

Welcome, Grid team. This is the letter that comes with the public
release of Nexus v0.1.0.

## TL;DR

One image on `ghcr.io/fun-fx/ffx_nexus:v0.1.0`. Pull it, point it at a
Postgres + ClickHouse + Redis, open `http://localhost:8081`, sign up
with your email, paste any provider key (Gemini, OpenAI, Anthropic,
**The Grid**), and your first chat completion is on the *Playground*
tab.

## What's in the box

- Single Go container at `ghcr.io/fun-fx/ffx_nexus:v0.1.0`. The
  dashboard ships embedded — no separate static host.
- OpenAI-compatible chat completions + streaming at `:8080/v1`.
- A console (:8081) with:
  - Welcome → Sign in/up → **Playground** (one-shot chat) → **Overview**
    (live traces, eval scores, model routing) → **Account** (your BYOK
    keys, virtual keys, monthly budget toggle).
  - Admin-only **Audit** tab for user/credential/key events.
- ClickHouse records every trace; the heuristic PII + completeness
  evaluators run inline, an async SLM judge (Ollama / vLLM) and a
  Python DeepEval+RAGAS sidecar run out-of-band.

## How to run

### 1. Minimal all-in-one (recommended for first try)

```bash
docker run --rm -p 8080:8080 -p 8081:8081 \
  -e NEXUS_POSTGRES_URL=postgres://nexus:nexus@host:5432/nexus?sslmode=disable \
  -e NEXUS_CLICKHOUSE_URL=clickhouse://nexus:nexus@host:9000/nexus \
  -e NEXUS_REDIS_URL=redis://host:6379/0 \
  -e NEXUS_MASTER_KEY="$(openssl rand -base64 32)" \
  -e NEXUS_ALLOW_SIGNUP=true \
  ghcr.io/fun-fx/ffx_nexus:v0.1.0
```

(Replace `host` with the canonical address of your datastore services.
If you'd rather use the bundled dev compose, run `docker compose -f
deploy/docker-compose.yml up -d postgres clickhouse redis` first and
then point `host` at `host.docker.internal`.)

### 2. From source

```bash
curl -fsSL https://raw.githubusercontent.com/fun-fx/ffx_nexus/v0.1.0/scripts/install.sh | bash
# (since v0.1.0 the repo is public — the raw github URL works)
```

The installer clones v0.1.0, brings up the local dev stack, builds the
Go binary, and starts the gateway on `:8090` / console on `:8091`.

### 3. One-shot on Kubernetes

```bash
helm install nexus oci://ghcr.io/fun-fx/ffx_nexus/charts/nexus \
  --version 0.1.0 \
  --set global.masterKey="$(openssl rand -base64 32)" \
  --set global.allowSignup=true
```

## Your first 5 minutes

1. Open `http://localhost:8081`.
2. Click **Sign in → Create account**, enter your email and a password
   (≥ 8 chars). You will be issued a virtual key (`nxs_live_...`) —
   copy it.
3. In **Account → My provider keys (BYOK)**, paste a real provider
   key (Gemini is fastest to test). The dashboard stores it encrypted
   under its own KEK.
4. Switch to **Playground**. Paste your virtual key once (it stays in
   `sessionStorage` for the rest of the tab). Pick a model, send a
   prompt.
5. Switch to **Overview**. Within five seconds your trace appears in
   **Recent traces** with the resolved model, latency, tokens, cost,
   and any cache/guardrail badges.

## What changes from the v0.4 design doc

- `NEXUS_KEY_MODE` default is now `strict_byok`. If you previously ran
  a setup that depended on env provider keys for *everyone*, set
  `NEXUS_ALLOW_SHARED_KEYS=true` to preserve the old behavior. No code
  changes needed otherwise.
- The signup flow no longer asks for a provider key at signup. The
  account-created confirmation now points at *My provider keys (BYOK)*
  as the next step. Existing signup flows that included a provider key
  still work — they just create the credential on signup.
- The first dashboard paint is the *Welcome* / *Sign-in* panel. You no
  longer see an empty operator console. Logged-in users start at
  Overview, with *Playground* as a second tab.

## What we expect from this pilot

We expect:

- (Reliability) The container stays healthy under modest traffic (a
  handful of users, ≤ 100 req/min each) on a single ClickHouse +
  Postgres + Redis. If you see OOM / 5xx in logs, please file a
  reproduction.
- (Telemetry) The dashboard polling is 5 s; latency cards and p95 are
  the canonical SLO surfaces.
- (BYOK) Each user pays their own provider bill. The operator
  container never makes an upstream call on behalf of the user when
  `NEXUS_ALLOW_SHARED_KEYS` is unset.
- (Routing) `model: "auto"` blends rolling eval quality, cost, and
  latency across all providers registered for that user. With one
  provider the routing table shows one row; with multiple it shows
  one row per provider/model.

## Known limitations

- The Grid provider is not in the registry until a user adds a BYOK
  key for it. Same goes for any other provider — this is by design.
- The Python eval service is optional. The console continues to work
  and the heuristic evals continue to feed routing even without it.
- Strict-byok requires Postgres + `NEXUS_MASTER_KEY`. Without storage
  the gateway falls back to `shared` and logs a warning.
